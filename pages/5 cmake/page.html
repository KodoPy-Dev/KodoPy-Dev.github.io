CMake, a scriptable build system

Hello and Welcome to the CMake section. In this CMake section you will find mostly a combination of different technologies all coming together. You can think of CMake as a scripting language that describes how you want to build your project.

Here is a short list of what CMake can do.

  • Generate cross-platform build files for various compilers (MSVC, GCC, Clang, etc.)
  • Manage dependencies and third-party libraries
  • Configure and control the build process with CMakeLists.txt
  • Support multiple build systems (Makefiles, Ninja, Visual Studio projects, etc.)
  • Enable out-of-source builds to keep source directories clean
  • Integrate with package managers like Conan and vcpkg
  • Handle complex project structures with subdirectories and modules
  • Configure for Debug and Release
  • Setup Test for TDD
  • Create installers for the end user
  • There are many builtin variables to easily configure your project

Integration

CMake is fully integrated into several code editors (such as VS-Code) and integrated development environments (IDEs) like Visual Studio. When it comes to VS-Code you can install the official CMake Extension CMake Extension. The CMake extension will let you navigate the project's build structure with a graphical interface, create build configurations, set breakpoints and debug CMake scripts, and really its a long list of features.

Concept

Of course, you should always check the Official CMake Docs for detailed information. But from a high-level perspective, CMake works by using configuration files called CMakeLists.txt within your project directory. These files are where you script your CMake commands. You typically place one CMakeLists.txt file in each directory of your project, with a main one at the root level, which serves as the entry point for the build configuration. The root level CMakeLists.txt file is where you would set the CMake version requirements and define your main project.

CMake is based around the concept of targets. A target is essentially an output unit of your build, such as an executable, a library, or even a test. The first step is defining your project by calling project(), where you specify your project's name. This defines the "root" target for your project. After that, you can create additional targets like **libraries** or **executables**, using commands such as add_library() and add_executable().

You can link these targets together using the target_link_libraries() command, allowing you to organize your project's structure and dependencies in a modular way. CMake also provides various target types, such as **STATIC**, **SHARED**, or **INTERFACE** libraries, giving you the flexibility to build your project in different configurations. In short, CMake allows you to build a project with various targets and link them in the most efficient and logical way for your needs.

By using this structure, CMake enables a clean, flexible, and scalable build process that works across different platforms and compilers. This modularity also makes it easier to manage large projects with multiple dependencies.

Common CMake Commands

CMake Build System

Command Category Description
project Project Defines the project name and optionally the version.
add_executable Targets Creates an executable target from the specified source files.
add_library Targets Creates a library target (static, shared, or module).
add_subdirectory Project Structure Adds a subdirectory containing a CMakeLists.txt to the build.
target_link_libraries Linking Links libraries to a target with appropriate visibility.
target_include_directories Includes Specifies include directories for a target.
find_package Dependencies Finds an installed package and makes it available.
find_library Dependencies Finds a library file and sets its path to the specified variable.
find_path Dependencies Finds a directory containing a specific file (e.g., a header).
find_program Dependencies Finds an executable program and sets its path to the specified variable.
set Variables Defines or modifies a CMake variable.
option Configuration Defines a boolean option that can be toggled by the user.
add_compile_definitions Compilation Adds preprocessor definitions globally.
target_compile_definitions Compilation Adds preprocessor definitions to a specific target.
target_compile_options Compilation Specifies additional compiler options for a target.
install Installation Defines installation rules for targets.
configure_file File Management Processes a file and optionally replaces variables before copying.
file File Management Finds files matching a pattern and stores them in a variable.
execute_process Execution Runs an external command during CMake execution.
message Debugging Prints a message to the console with different severity levels.

Examples

These examples do not demonstrate all the possible argument options!

Any of the commands that have (source.cpp) being passed in as an argument can take a list of sources. Like this, add_executable(MyApp source_a.cpp source_b.cpp source_c.cpp) 

# Define CMake Min Version (Check the version number of CMake that you are using)
cmake_minimum_required(VERSION 4.0)

# Define a Project (Add Version number if desired)
project(MyProject VERSION 1.2.3)

# Get the version number info if defined by project command
# ${PROJECT_VERSION} → "1.2.3"
# ${PROJECT_VERSION_MAJOR} → "1"
# ${PROJECT_VERSION_MINOR} → "2"
# ${PROJECT_VERSION_PATCH} → "3"

# Create Executable (Creates a .exe file)
add_executable(MyApp source.cpp)

# Create Static Lib (Used for linking at compile time)
add_library(MyLib STATIC source.cpp)

# Create Dynamic Lib (Used for linking at run time)
add_library(MyLib SHARED source.cpp)

# Link a Static Lib to and Executable
add_library(MyLib STATIC source.cpp)
add_executable(MyApp source.cpp)
target_link_libraries(MyApp MyLib)

# Add Header Files
add_library(MyLib STATIC source.cpp)
target_include_directories(MyLib INTERFACE ${CMAKE_SOURCE_DIR}/include)
